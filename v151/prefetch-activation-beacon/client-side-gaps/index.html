Why not client-side JavaScript?

The explainer rejects sendBeacon-on-load reporting as unreliable. Here is the problem, live: a naive client-side "activation" counter driven by real page-lifecycle events drifts away from the truth as you interact.

The true activation count for one page view is 1. Watch a naive client-side reporter over-count as the tab is hidden/shown and restored from the back/forward cache — the very failures the browser beacon avoids by firing once, from the navigation stack.

The three failure modes (from the explainer)

  • Cache ambiguity — telemetry filters back/forward navigations to avoid double counting, so genuine bfcache activations of a prefetched page get missed.
  • Duplicate reporting — a back/forward-cache restore re-runs the page's scripts, so the client reporter fires again for a single real visit.
  • Race conditions — the script often does not run early enough to capture rapid transitions, so the signal is lost.

The on-prefetch-activation beacon sidesteps all three: the browser dispatches exactly one credentialless HEAD request at commit, before any page script or service worker runs.

How the naive reporter is wired

// A typical (unreliable) client-side reporter:
addEventListener("pageshow", (e) => {
  // fires on bfcache restore too (e.persisted === true) → double count
  navigator.sendBeacon("/telemetry/activation");
});
document.addEventListener("visibilitychange", () => {
  if (document.visibilityState === "visible") navigator.sendBeacon("/telemetry/activation");
});
