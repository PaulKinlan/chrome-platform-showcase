WebRTC Data Channel: SCTP Negotiation Acceleration Protocol

WebRTC data channels run SCTP over a DTLS association. The standard SCTP handshake costs extra round trips before the first channel can open. SNAP (SCTP Negotiation Acceleration Protocol) embeds the SCTP INIT parameters directly in the SDP offer/answer as a new a=sctp-init media attribute, cutting up to two network round-trip times off data-channel establishment. The JavaScript surface — RTCPeerConnection and RTCDataChannel — is unchanged; SNAP is negotiated transparently in the SDP.

concepts

  1. Open-Latency Lab

    Run a real same-page RTCPeerConnection loopback, open a data channel, and time createDataChannel → onopen across repeated runs. This is exactly the interval SNAP shortens — you measure the live establishment cost SNAP targets.

  2. SDP & SCTP Inspector

    Generate a real offer with createOffer() and dissect the SCTP negotiation lines it carries: the m=application … UDP/DTLS/SCTP line, a=sctp-port, a=max-message-size, a=setup, a=fingerprint — and probe for the SNAP a=sctp-init attribute itself, honestly reporting present or absent.

  3. Reliable vs Unreliable Streams

    Open several data channels over one loopback with different SCTP reliability settings — ordered, maxRetransmits, maxPacketLifeTime, and pre-negotiated id — then send messages and watch ordering and delivery differ. These are the SCTP association parameters SNAP negotiates up front.

why it shipped

For real-time apps — game state, collaborative cursors, file transfer signalling — the time to open the first data channel is user-visible latency. The classic flow needs the DTLS handshake and then a separate SCTP INIT/INIT-ACK/COOKIE exchange before any application data flows. SNAP piggybacks the SCTP initialization parameters onto the SDP that both peers already exchange through signalling, so the SCTP association is effectively pre-agreed by the time DTLS completes. The draft reports savings of up to two round trips, with no change required to application code.
