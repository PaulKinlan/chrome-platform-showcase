Open-Latency Lab

Open a real data channel over a same-page RTCPeerConnection loopback and time createDataChannel → the channel's open event. That interval — the DTLS plus SCTP association setup — is exactly what SNAP shortens by pre-negotiating the SCTP INIT parameters in SDP.

Both peers live in this one page, so ICE resolves over host candidates with near-zero network delay. The absolute numbers are a floor, not real-world latency — but the run-to-run measurement is genuine, and it isolates the establishment work SNAP removes. The RTCPeerConnection / RTCDataChannel API is unchanged by SNAP; nothing here is simulated.

how it works

Each run builds two RTCPeerConnections, opens a data channel on the offerer, and shuttles the offer, answer, and ICE candidates between them directly (no signalling server needed for loopback). We start a high-resolution timer at createDataChannel() and stop it in the channel's open event:

const t0 = performance.now();
const a = new RTCPeerConnection();
const b = new RTCPeerConnection();
a.onicecandidate = (e) => e.candidate && b.addIceCandidate(e.candidate);
b.onicecandidate = (e) => e.candidate && a.addIceCandidate(e.candidate);

const dc = a.createDataChannel("probe");   // SCTP association starts here
dc.addEventListener("open", () => {
  const openMs = performance.now() - t0;    // ← SNAP targets this interval
});

const offer = await a.createOffer();
await a.setLocalDescription(offer);
await b.setRemoteDescription(offer);
const answer = await b.createAnswer();
await b.setLocalDescription(answer);
await a.setRemoteDescription(answer);

On a real network the SCTP INIT/INIT-ACK/COOKIE-ECHO/COOKIE-ACK exchange happens after DTLS finishes, adding up to two round trips. SNAP carries those INIT parameters in the SDP you already exchange, so the association is ready as soon as DTLS is — the establishment interval this lab measures is what shrinks.
