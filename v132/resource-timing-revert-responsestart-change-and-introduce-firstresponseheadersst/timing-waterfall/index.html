Timing Waterfall

Chrome 132 introduces firstResponseHeadersStart to the Resource Timing API — a new timestamp marking when the browser received the first byte of response headers. This is more precise than responseStart (which could be early hints). Fetch a resource below and compare all timing fields in a live waterfall.

// Chrome 132+ — use firstResponseHeadersStart for TTFB
const [entry] = performance.getEntriesByName('/api/data');
const ttfb = entry.firstResponseHeadersStart - entry.requestStart;
//             ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

//             responseStart might be early hints (103 status)

// responseStart — may be from an early hints (103) response
// firstResponseHeadersStart — always the actual response headers

// Key timing phases:
const phases = {
  dns:       entry.domainLookupEnd  - entry.domainLookupStart,
  tcp:       entry.connectEnd       - entry.connectStart,
  ttfb:      entry.firstResponseHeadersStart - entry.requestStart,
  headers:   entry.responseStart    - entry.firstResponseHeadersStart,
  body:      entry.responseEnd      - entry.responseStart,
};
