Request.isReloadNavigation

Chrome 149 adds a read-only boolean isReloadNavigation to the Fetch API's Request interface. In a service worker's fetch event handler, this flag is true when the user explicitly reloaded the page — letting service workers apply different caching strategies for reloads versus normal navigations.

concepts

  1. Service Worker Demo

    Registers a service worker that logs request.isReloadNavigation for each navigation request. Navigate to the page normally, then press Cmd/Ctrl+R (or F5) to reload — the flag switches, and the service worker can respond differently for each case.

  2. API Explorer

    Read isReloadNavigation from a Request object directly in the page (non-navigation context) and understand when it can be true. Includes a reference of the full attribute spec and comparison with the existing Request.mode / Request.destination fields.

why it shipped

Service workers routinely need to distinguish a fresh navigation from a user reload. A reload is an explicit user signal that they want up-to-date content — so a service worker that normally serves a cached shell for speed should probably bypass the cache and fetch fresh HTML on reload. Before this attribute, the only proxy was inspecting request.headers.get('Cache-Control') for no-cache (which browsers send on hard reload) or cross-referencing navigator.appVersion — both fragile. isReloadNavigation gives a precise, first-class signal directly on the Request object where service workers already do their routing logic.

reload types

  • Normal navigation (link click, address-bar enter, JS location.href) — isReloadNavigation is false
  • Soft reload (Cmd/Ctrl+R, F5) — isReloadNavigation is true
  • Hard reload (Cmd/Ctrl+Shift+R, Shift+F5) — isReloadNavigation is true (the existing Cache-Control: no-cache header also signals hard reload)
  • Non-navigation requests (subresources, API calls) — isReloadNavigation is always false
