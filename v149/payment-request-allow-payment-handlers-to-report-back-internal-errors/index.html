v149 · Web APIs · Payment Request

Allow payment handlers to report back internal errors

Chrome 149 lets payment handler service workers report specific internal errors back to the merchant page via the Payment Request API. Previously a handler could only succeed or generically abort; now it can surface a structured error message that the merchant can inspect and display.

why it shipped

The Payment Request API allows merchant sites to invoke registered payment handlers (browser built-ins, or third-party service worker handlers). When a handler encountered an internal error — a failed 3DS challenge, an expired card, a network timeout with the payment network — the only option was to call PaymentRequestEvent.respondWith(Promise.reject()), which surfaced as a generic AbortError on the merchant side. Chrome 149 adds a mechanism for the handler to include a structured error detail that survives the boundary and reaches the merchant's catch block.

the change

// Payment handler service worker (before Chrome 149)
self.addEventListener('paymentrequest', event => {
  event.respondWith(Promise.reject(new Error('Card expired')));
  // → merchant only sees DOMException: AbortError, no detail
});

// Payment handler service worker (Chrome 149+)
self.addEventListener('paymentrequest', event => {
  event.respondWith(
    Promise.reject({
      error: 'card_expired',
      message: 'Your card expired 2024-01. Update your payment method.'
    })
  );
});

// Merchant page (Chrome 149+)
try {
  const response = await request.show();
  await response.complete('success');
} catch (err) {
  // err.detail is the object the handler rejected with
  if (err.detail?.error === 'card_expired') {
    showCardUpdateFlow(err.detail.message);
  }
}
