Clip Text overflow on user interaction

Chrome 149 improves editing and caret-navigation in elements with text-overflow: ellipsis. When a user interacts with such an element — placing a caret or editing — the browser temporarily switches from ellipsis to clip, revealing the full text. When focus leaves, ellipsis returns.

concepts

  1. Editing Demo

    A contenteditable element with text-overflow: ellipsis. Click into it to edit — the full text reveals, the ellipsis disappears. Click away to see the clipped display snap back. No JavaScript required.

  2. Caret Navigation Demo

    Tab into an overflow-clipped element to position the caret. The browser unclips the text so you can navigate to hidden characters with arrow keys. A comparison shows the old clipped-while-focused behaviour.

why it shipped

Before Chrome 149, text-overflow: ellipsis was applied regardless of whether the element was being edited or had a caret inside it. This caused two serious usability problems: when editing a truncated field, users could only see and interact with the visible portion (text beyond the ellipsis was invisible even as they typed or deleted); and keyboard navigation with arrow keys would move the caret into the hidden region without the user being able to see where the cursor was. Chrome 149 makes the browser aware that interaction changes the rendering context — the ellipsis is suppressed for the duration of focus or caret presence, matching the intuitive expectation that "I can see what I'm editing".

how it works

  • No CSS or JavaScript change required — the behaviour is automatic
  • Applies to any element with text-overflow: ellipsis that receives focus or a caret placement (via editing or keyboard navigation)
  • When the interaction ends (blur, caret removed), the element reverts to showing the ellipsis
  • Works in contenteditable elements, <input>, and <textarea> with text-overflow
