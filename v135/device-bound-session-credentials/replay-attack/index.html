Cookie replay attack visualised

DBSC's headline use case from the Chrome blog post: session cookie theft no longer hands the attacker the session. The simulation has three actors — the real user, the server, and an attacker who exfiltrated the cookie. Watch the server verify (or reject) the replayed cookie depending on which actor signs the challenge.

Flag DBSC ships behind chrome://flags/#device-bound-session-credentials. The simulator below uses WebCrypto's ECDSA P-256 directly to show the cryptographic shape the browser would use under the hood.

the protocol

// Real flow happens in HTTP headers.
Sec-Session-Registration: alg=ES256
Sec-Session-Response: { jwk: <public>, sig: <reg-proof> }

// On every refresh, server issues a fresh challenge.
Sec-Session-Challenge: <nonce>
Sec-Session-Response: sig(nonce, bound-priv-key)

// Server verifies sig with the bound public key.
// Stolen cookie + attacker's key → signature fails → 401.
