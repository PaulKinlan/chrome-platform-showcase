Step through the full DBSC session lifecycle: initial registration, short-lived token issuance, automatic token rotation before expiry, and what happens when a token is stolen and replayed from another device. Configurable rotation interval and theft scenario.

checking DBSC availability…
DBSC works by binding a short-lived session token to a device-local private key. The browser automatically refreshes the token before it expires by signing a challenge with that key. A stolen token is worthless: the attacker can't sign the next refresh challenge because they don't have the private key. This demo simulates the timeline so you can see exactly when rotation fires and why replay fails.

// Server-side token verification (simplified) function verifyRequest(req) { const token = req.cookies['session']; const proof = req.headers['Sec-Session-Response']; // 1. Token bound to a registered public key const binding = db.getBinding(token.sessionId); if (!binding) return 401; // Unknown session // 2. Short-lived — reject if expired if (token.exp < Date.now()) return 401; // ← stolen stale token fails here // 3. Proof-of-possession: browser signed challenge with device key if (!verify(proof, challenge, binding.publicKey)) return 401; // ← stolen token on // another device fails here return 200; // ✓ Bound to this device, not expired }
