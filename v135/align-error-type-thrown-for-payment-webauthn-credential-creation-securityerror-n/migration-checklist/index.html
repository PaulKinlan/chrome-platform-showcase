Migration checklist for payment credential errors

The historical inconsistency: WebAuthn credential creation for payment credentials in a cross-origin iframe without user activation threw NotAllowedError (the spec's payment-specific behaviour), but the rest of WebAuthn throws SecurityError in the same situation. Chrome 135 aligns the error type. If your try/catch branched on the old name, you need to update. The linter below scans your code for the bad pattern.

old: NotAllowedError new: SecurityError

before Chrome 135

try {
  await navigator.credentials.create({ ... payment ... });
} catch (e) {
  if (e.name === "NotAllowedError") {
    /* user dismissed OR cross-origin OR no activation */
  }
}

Chrome 135+

try {
  await navigator.credentials.create({ ... payment ... });
} catch (e) {
  if (e.name === "SecurityError") {
    /* cross-origin OR no activation */
  } else if (e.name === "NotAllowedError") {
    /* user dismissed */
  }
}
