Fetch Intercept Demo

Before Chrome 135, a srcdoc iframe was not a Service Worker client — fetches it made bypassed the SW entirely. This demo creates an iframe srcdoc, makes a fetch from inside it, and shows whether the Service Worker intercepted it. Compare the SW-controlled vs uncontrolled case.

checking Service Worker… checking srcdoc SW client…
Chrome 135 creates proper Service Worker clients for srcdoc iframes and makes them inherit the parent's SW controller. Before this fix, a srcdoc iframe's fetches appeared in Resource Timing but the SW never saw them — an inconsistency that broke offline-first apps and fetch interceptors. After Chrome 135, the SW can intercept, cache, and modify responses from srcdoc content.

// sw.js — fetch interception now covers srcdoc iframes (Chrome 135+) self.addEventListener('fetch', event => { const url = new URL(event.request.url); console.log('[SW] Fetch intercepted from:', event.clientId); // Chrome 135: event.clientId now resolves for srcdoc clients! // You can now cache srcdoc iframe requests: event.respondWith( caches.match(event.request).then(cached => { if (cached) return cached; return fetch(event.request).then(resp => { // Cache for offline use const clone = resp.clone(); caches.open('v1').then(c => c.put(event.request, clone)); return resp; }); }) ); });
