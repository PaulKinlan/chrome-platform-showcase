Component encapsulation with anchor-scope

The planning use case wasn't just "lists with duplicate anchor names". It was components: a reusable date-picker, menu, or popover that hard-codes an internal anchor-name needs to make sure a second copy on the page doesn't steal anchors from the first. anchor-scope on the component root is the encapsulation primitive.

Toggle the scope. With anchor-scope on each component, B's popover anchors to B's trigger. Without it, both popovers see both triggers — the cascade picks one and both popovers stack on top of it.

the css

/* component root contains the anchor name */
.component {
  anchor-scope: --menu-anchor;
}

/* internal trigger declares the name */
.menu-trigger { anchor-name: --menu-anchor; }

/* internal popover references it */
.menu-popover {
  position: absolute;
  position-anchor: --menu-anchor;
  top: anchor(bottom);
  left: anchor(left);
}

contrast with the list demo

The companion list demo illustrates the bug at row scope. This one is the canonical fix at component scope — what the spec authors had in mind. A design system can ship a popover web-component, a tooltip, an autocomplete, all using internal anchor names without leaking them across instances.
