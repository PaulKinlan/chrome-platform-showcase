Source vs Trigger response headers — what changed

The Attribution Reporting API has two distinct HTTP response headers: Attribution-Reporting-Register-Source (from the ad click) and Attribution-Reporting-Register-Trigger (from the conversion page). v131's change applies only to the latter. This page lays out concrete header values, showing where the 25-char cap still applies and where it doesn't.

1. source registration (ad impression)

Attribution-Reporting-Register-Source
{
  "source_event_id": "12345",
  "destination": "https://shop.example",
  "aggregation_keys": {
    "campaignCounts": "0x159",
    "geoValue":        "0x5"
  }
}
aggregation key names still capped at 25 chars (persisted)

2. trigger registration (conversion)

Attribution-Reporting-Register-Trigger
{
  "aggregatable_trigger_data": [{
    "key_piece": "0x400",
    "source_keys": [
      "campaign_id_for_holiday_2026_q4_promo_long_name_yes"
    ]
  }],
  "aggregatable_values": {
    "campaign_id_for_holiday_2026_q4_promo_long_name_yes": 1664
  }
}
v131: key identifiers can be any length

why the asymmetry

  • Sources are persisted on disk for the source-expiry window (default 30 days). Every byte counts against the browser's storage quota for the user — the cap protects user storage.
  • Triggers are matched and reported in-process. They never persist as plain identifiers. So the cap was only ever a redundant validation step. Removing it lets ad-tech use descriptive identifiers without truncation.
