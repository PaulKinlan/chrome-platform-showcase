Bounce trackers inject identifiers via the HTTP cache during redirect chains. Chrome 134's mitigation intercepts the request at each bounce hop and prevents cache-based ID persistence. Pick a scenario and trace what happens at each hop.

Chrome 134+ required for mitigations
A bounce tracker works by routing you through tracker.example/?id=user123 — a domain it controls — before sending you to your destination. The tracker's cache stores a unique response keyed to your ID, building a cross-site profile without setting cookies. Chrome 134 applies HTTP cache partitioning and storage clearing to break this chain.

Why HTTP cache?

Cookies, localStorage, and IndexedDB are partitioned by top-level site in Chrome. But historically, the HTTP cache was shared — a tracker could check: "did I cache a response for this user's IP/config combo?" to correlate visits. Cache partitioning in Chrome 86+ partitioned the cache by (top-level site, frame site), and Chrome 134 extends bounce-tracking mitigations to also clear cache storage for classified trackers, closing this loophole.
