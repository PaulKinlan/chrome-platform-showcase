DBSC Demo

Shows the registration handshake headers for Device Bound Session Credentials and checks for Chrome 145 DBSC support indicators.

DBSC is a server-driven protocol — it requires a cooperating server that sends Sec-Session-Registration headers. This demo shows the expected header exchange and checks for browser support.

server registration header

# Step 1: Server sends Sec-Session-Registration in login response
HTTP/1.1 200 OK
Set-Cookie: session=abc123; Secure; HttpOnly; SameSite=Strict; Max-Age=30
Sec-Session-Registration: path="/dbsc/register";
    challenge="dGhpcyBpcyBhIGNoYWxsZW5nZQ==";
    authorization="https://auth.example/dbsc"

# Step 2: Chrome generates a device-local EC P-256 key pair
# Chrome registers the session by POSTing to the registration path
POST /dbsc/register HTTP/1.1
Cookie: session=abc123
Content-Type: application/json

{
  "proof": "eyJhbGciOiJFUzI1NiIsImtleSI6ey4uLn19..."
}
# The proof is a signed JWT: header contains the device public key,
# body contains the challenge; signed with device private key.

# Step 3: Server stores public key, issues long-term session
HTTP/1.1 200 OK
Set-Cookie: session=xyz789; Secure; HttpOnly; SameSite=Strict

periodic refresh challenge

# When short-lived session expires, server challenges the browser:
HTTP/1.1 401 Unauthorized
Sec-Session-Challenge: challenge="newChallengeBase64";
    id="session-id-123"

# Chrome responds with a signed proof using the stored private key:
GET /resource HTTP/1.1
Sec-Session-Response: eyJhbGciOiJFUzI1NiIsImtleSI6ey4uLn19...

# If the device private key is gone (stolen cookie, different device):
# The signed response is missing or invalid → server invalidates session
