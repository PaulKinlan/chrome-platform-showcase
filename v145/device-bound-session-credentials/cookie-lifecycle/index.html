Cookie lifecycle simulator

Step through a full DBSC handshake — short-lived cookie, refresh challenge, signed proof-of-possession, fresh cookie — and watch what each request carries. Helps you reason about exactly which moments are device-bound.

Behind a flag. Enable chrome://flags/#enable-device-bound-session-credentials to see a real handshake with the browser's TPM-backed key. The simulator below runs the same state machine in JavaScript.

state machine

// 1. Login: server returns Sec-Session-Registration + cookie
// 2. Browser generates device-bound keypair, posts public key + JWT proof
// 3. Server binds the session ID to the public key, returns short-lived cookie
// 4. Cookie expires; before next request, browser silently:
//    - fetches /session/refresh
//    - server returns a challenge nonce
//    - browser signs nonce with bound private key (no JS sees it)
//    - server verifies and issues a fresh short-lived cookie
// Result: even if cookie is stolen, attacker cannot refresh it
// from another device — they don't have the private key.
