Preferred Capture

Click capture to run a real getDisplayMedia() call that asks for audio and adds the audioSelection: "preferred" hint. The page then reports the audio track the user agent actually attached — its label, getSettings(), and a live level meter — so you can see whether audio came along.

Hint, behind a flag. audioSelection is a hint — you still choose in the picker. For it to take effect, enable it in Chrome 152+: open chrome://flags/#enable-experimental-web-platform-features and set Enabled, or launch Chrome with --enable-blink-features=GetDisplayMediaAudioSelection (or --enable-experimental-web-platform-features). Without it the member is silently ignored — getDisplayMedia() still runs, just without the audio-preference hint. A dictionary member like this cannot be feature-detected from JavaScript, so this demo reports the observable outcome (was an audio track attached?), not a claim that the hint fired.

the call

const stream = await navigator.mediaDevices.getDisplayMedia({
  video: true,
  audio: true,
  audioSelection: "preferred"   // NEW in 152 — prefer capturing audio too
});

const audio = stream.getAudioTracks()[0];
console.log(audio ? audio.getSettings() : "user shared video only");

Only value in AudioSelectionPreferenceEnum today is "preferred". It raises the priority of offering and pre-selecting audio in the picker; it never overrides the user's final choice, and it does not apply audio constraints (those still live in audio).
