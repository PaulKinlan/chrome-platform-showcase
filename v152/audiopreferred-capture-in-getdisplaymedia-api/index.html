audioPreferred capture in getDisplayMedia API

A new audioSelection member on DisplayMediaStreamOptions lets a screen-sharing app hint that it would prefer audio to be captured alongside the video. Its only value today is "preferred". It is a hint — the user still chooses in the picker — but it nudges the user agent to prioritise offering and pre-selecting audio.

concepts

  1. Preferred Capture

    The real call: getDisplayMedia({ video, audio: true, audioSelection: "preferred" }). Surfaces the audio track the user agent actually attached — label, getSettings(), and a live level meter proving real audio flows. Honest fallback when the hint or capture isn't available.

  2. Hint vs Default

    Run two captures — one with audioSelection: "preferred", one without — and record what each returned into a comparison table. Shows what the hint is for: raising the chance audio comes along, without ever forcing it.

  3. Options Inspector

    Build the DisplayMediaStreamOptions object by toggling audioSelection alongside its siblings — systemAudio, windowAudio, monitorTypeSurfaces — see the exact JSON that gets passed, then run it. Explains why a dictionary hint can't be feature-detected in JS and how to degrade honestly.

why it shipped

Apps that record or stream a screen — tutorials, remote support, presentation capture — frequently want the accompanying audio, but the getDisplayMedia picker historically defaulted to video-only and required the user to notice and tick an audio option. Missing that step produces silent recordings that only surface as a problem after the fact. audioSelection: "preferred" gives the application a way to signal its intent so the user agent can prioritise offering (and pre-selecting) audio in the picker, while keeping the final decision with the user. Because it is a dictionary member, browsers that don't implement it ignore it silently — the capture still works, just without the hint.
