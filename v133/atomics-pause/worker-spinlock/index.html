Real spinlock in a Worker

The TC39 proposal motivates Atomics.pause() with one scenario: a thread spin-waiting on a flag in SharedArrayBuffer. This page runs that exact pattern in a Worker, with and without the pause hint.

SharedArrayBuffer requires cross-origin isolation (COOP+COEP headers). This page won't be isolated on the showcase domain, so the worker will fall back to a non-shared run. The numbers will still print — just from a single-thread fake.

the pattern

Producer thread writes a value, then flips a shared flag. Consumer thread spin-waits on the flag, reads the value, flips it back. Atomics.pause() goes inside the spin — on x86 it issues the PAUSE instruction, which lowers core power use, reduces memory-ordering pipeline flushes when the flag flips, and on hyperthreaded cores yields execution resources to the sibling thread. Functionally a no-op, but the CPU schedules differently around it.

// Inside the worker (consumer)
function waitFor(flag, want) {
  while (Atomics.load(flag, 0) !== want) {
    Atomics.pause();   // hint: I'm spinning, back off
  }
}
