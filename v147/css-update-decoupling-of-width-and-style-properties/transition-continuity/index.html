← CSS update: decoupling of Width and Style properties

v147 · CSS · demo

Transition Continuity

When border-style changes alongside border-width, the old Chrome behavior snapped the width to 0 — killing smooth transitions. Chrome 147 fixes this by computing width independently. Toggle the boxes below to see the difference.

Chrome <147 — old behaviour
border-style toggles between solid ↔ none
border-width snaps to 0 — no animation
Chrome 147+ — decoupled
border-width animates 0 → 8px smoothly
even while border-style changes

Why does this happen?

In the old model, the CSS cascade computes border-width as 0px when border-style is none. So when a transition tries to animate from border-style:none; border-width:0px to border-style:solid; border-width:8px, both values change simultaneously — and since border-style cannot be interpolated, the transition for border-width also gets corrupted.

In Chrome 147+, border-width computes to the author-specified value (8px) even when border-style is none. The browser can now correctly interpolate the width from 0 to 8px while separately handling the non-animatable style change — resulting in a smooth animation.

/* Works as expected in Chrome 147+, Firefox, and Safari */
.box {
  border-width: 0;
  border-style: none;
  border-color: blue;
  transition: border-width 0.6s ease;
}
.box.active {
  border-width: 8px;
  border-style: solid;
  /* border-width now transitions smoothly: 0 → 8px */
}

/* Old Chrome workaround: keep border-style:solid always,
   use opacity or clip to hide the border visually */
.box-workaround {
  border-style: solid;
  border-color: transparent;
  border-width: 0;
  transition: border-width 0.6s ease, border-color 0s 0.3s;
}
.box-workaround.active {
  border-width: 8px;
  border-color: blue;
}
