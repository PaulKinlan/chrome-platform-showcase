Migration Guide

Practical steps for moving off inline XSLT → SVG rendering: how to identify affected URLs, and three migration paths ranked by complexity.

Most sites are not affected. Only sites that serve XML files with Content-Type: application/xml or text/xml that include an <?xml-stylesheet type="text/xsl"> processing instruction pointing to an XSLT that outputs xmlns="http://www.w3.org/2000/svg" root elements are impacted.

how to detect if you are affected

grep for the pattern in your XML files
# Look for xml-stylesheet processing instructions
grep -r 'xml-stylesheet.*type="text/xsl"' /path/to/xml/files/

# Check if those XSLT files produce SVG output
grep -l '<xsl:element name="svg"' /path/to/xsl/files/
grep -l 'xmlns="http://www.w3.org/2000/svg"' /path/to/xsl/files/

migration option 1: serve SVG directly (simplest)

Convert XML → SVG at build time

Run your XSLT transformation offline (using Saxon, libxslt, or similar) and commit the output as .svg files. Serve them with Content-Type: image/svg+xml. No client-side transformation needed.

# Build step: Saxon XSLT processor
java -jar saxon.jar -s:data.xml -xsl:transform.xsl -o:output.svg

# Or libxslt
xsltproc transform.xsl data.xml > output.svg

migration option 2: XSLTProcessor in JavaScript

Transform client-side with XSLTProcessor

XSLTProcessor is still available in Chrome 147 and performs the same transformation. The result can be inserted into the DOM as SVG.

// Fetch and transform XML using XSLTProcessor
const [xmlResp, xslResp] = await Promise.all([
  fetch('/data.xml'),
  fetch('/transform.xsl'),
]);
const [xmlText, xslText] = await Promise.all([
  xmlResp.text(), xslResp.text()
]);

const parser = new DOMParser();
const xmlDoc = parser.parseFromString(xmlText, 'application/xml');
const xslDoc = parser.parseFromString(xslText, 'application/xml');

const processor = new XSLTProcessor();
processor.importStylesheet(xslDoc);

const resultDoc = processor.transformToDocument(xmlDoc);
// Insert SVG result into the page
document.body.appendChild(
  document.adoptNode(resultDoc.documentElement)
);

migration option 3: server-side transform

Transform on the server, serve SVG

Run the XSLT transformation on the server for each request (or cache the result). The client receives SVG directly with Content-Type: image/svg+xml — no client-side transformation at all.

// Node.js example with node-xslt (uses libxslt under the hood)
const xslt = require('node-xslt');
const fs = require('fs');

app.get('/chart.svg', (req, res) => {
  const xml = fs.readFileSync('data.xml', 'utf8');
  const xsl = fs.readFileSync('transform.xsl', 'utf8');
  const svg = xslt.transform(xsl, xml);

  res.setHeader('Content-Type', 'image/svg+xml');
  res.send(svg);
});
