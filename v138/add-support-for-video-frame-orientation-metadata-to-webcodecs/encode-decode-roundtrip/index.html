Encode → decode round-trip

The motivation case from the spec: an Android camera produces a portrait frame, the container records orientation as metadata, but a decoded VideoFrame historically had no way to carry that. Run the encoder, run the decoder, see whether rotation + flip survive the trip.

the trick

// 1. wrap the captured pixels with orientation metadata
const sourceFrame = new VideoFrame(srcCanvas, {
  timestamp: 0,
  rotation: 90,              // NEW in 138
  flip: "horizontal",        // NEW in 138
});

// 2. encode it
encoder.encode(sourceFrame);

// 3. on the decoder side, the new VideoFrame carries
//    the same metadata through to the renderer.
const decoder = new VideoDecoder({
  output: (frame) => {
    console.log(frame.rotation, frame.flip); // 90, "horizontal"
    frame.close();
  },
  error: console.error,
});

Before Chrome 138, the only way to carry orientation across the encode/decode boundary was a separate JS sidecar — awkward to wire, easy to lose during a worker hop or a transmuxed format. Now the metadata travels on the VideoFrame itself.
