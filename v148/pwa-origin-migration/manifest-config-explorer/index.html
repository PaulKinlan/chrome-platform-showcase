Manifest Config Explorer

See exactly which fields go in the old origin's manifest and which go in the new one — and how the browser uses both to validate and execute the migration.

Old origin app.oldexample.com 
{
  "name": "My Great App",
  "short_name": "GreatApp",
  "start_url": "/",
  "display": "standalone",

  "migration": {
    "new_origin":
      "https://app.newexample.com"
  }
}
New origin app.newexample.com 
{
  "name": "My Great App",
  "short_name": "GreatApp",
  "start_url": "/",
  "display": "standalone",

  "migration": {
    "old_origin":
      "https://app.oldexample.com"
  }
}
browser verifies both claims before migrating

Field reference

Field Origin Role Notes
migration.new_origin Old origin's manifest Declaration of intent "I want to move to this URL"
migration.old_origin New origin's manifest Confirmation of claim "I accept the migration from that URL"
name / short_name Both Continuity signal Should match for a smooth UX (no rename prompt)
id New origin Optional override Omit to inherit old app identity; set to create new identity

Validation rules

  • Mutual claim required — both manifests must declare the relationship. One-sided declarations are ignored.
  • Same-site constraint — old and new origins must share a registrable domain (e.g. example.com). Cross-site migrations are not permitted.
  • HTTPS only — both origins must be served over HTTPS.
  • App already installed — migration only applies to users who have the old origin's PWA installed; new users install normally from the new origin.
// Old origin service worker — signal readiness
self.addEventListener('install', () => self.skipWaiting());

// The browser reads manifest.migration.new_origin on the old origin.
// When the user next launches the app, Chrome checks new_origin for
// a matching manifest.migration.old_origin declaration.
// If both are present and the same-site check passes, the installed
// app entry silently updates to point to the new origin.
