Notification attribution for PWAs on macOS

Installed PWAs on macOS get notifications attributed to the app itself — its name and icon in Notification Center — instead of "Google Chrome". Two web-observable behaviour changes ride along, matching WebKit: requireInteraction is no longer honoured for app notifications (macOS users control persistence per app), and the Badging API now requires notification permission before a badge shows. The attribution itself is OS UI — these demos probe what can be probed, make the real API calls, and label the macOS-only rendering as a before/after comparison rather than pretending to detect it.

concepts

  1. Attribution before & after

    A rendered comparison of the same push notification attributed to Chrome vs to the installed PWA, beside a real environment probe: platform, display-mode: standalone state, Notification API presence and current permission.

  2. requireInteraction tester

    Fires a real new Notification(…) with requireInteraction on or off and reports exactly what this browser did — including why the flag is now ignored for attributed app notifications on macOS.

  3. Badge permission gate

    Calls the real navigator.setAppBadge() / clearAppBadge() and pairs the result with the live Notification.permission state — the new prerequisite on macOS. The API resolves either way; the badge only renders once permission is granted.

why it shipped

A PWA that a user deliberately installed should feel like an app, not like a Chrome tab that occasionally shouts. Attributing notifications to Chrome muddled the mental model ("why is Chrome notifying me about my chat app?"), buried per-app notification controls in browser settings instead of System Settings, and diverged from what Safari-installed web apps already did. Aligning with WebKit also meant adopting macOS conventions: the OS, not the site, decides whether a notification is persistent (killing requireInteraction for app notifications), and a dock badge is part of the same per-app notification grant. Enterprises pre-granting permission via NotificationsAllowedForUrls now also need an MDM profile for the PWA's bundle ID.
