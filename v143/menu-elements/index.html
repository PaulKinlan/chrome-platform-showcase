Menu elements

Native application menus for the web: <menubar>, <menulist>, <menuitem> and <submenu> give you File/Edit/View-style menus with the correct implicit ARIA roles (menubar, menu, menuitem), popover + anchor positioning built in, and the APG keyboard model out of the box — no more re-implementing the hardest ARIA pattern by hand. Behind --enable-blink-features=MenuElements in the developer trial; every demo probes the real element upgrade and runs the real thing when the flag is on.

concepts

  1. Application menubar

    A real <menubar> with nested <submenu> flyouts, separators and a disabled item. Probes whether this browser upgrades the elements (constructor + implicit role) and falls back to clearly-labelled inert markup when it doesn't.

  2. Button-invoked menulist

    The one-off dropdown case: <button command="toggle-menu" commandfor="…"> opening a standalone <menulist> popover, with the implicit aria-haspopup=menu / aria-expanded wiring read back live from the DOM.

  3. Checkable menu items

    State in menus: <fieldset checkable="single"> radio groups and checkable="multiple" toggles with defaultchecked, with a live readout of each item's real checked value as you click.

why it shipped

The existing <menu> element is a false friend: despite its name it maps to a plain list, not to the ARIA menu/menubar pattern, so every product that needs an application menu rebuilds focus management, typeahead, checkable state and submenu positioning from scratch — and most get it subtly wrong. The Open UI menu elements proposal layers real menu semantics on top of two things the platform already shipped: popovers (for the flyout behaviour) and invoker commands (for the button-opens-menu wiring). Chrome 143 puts the elements into developer trial to gather feedback on the markup shape, the <fieldset checkable> grouping model and the keyboard contract.
