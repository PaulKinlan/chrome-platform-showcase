Replacement text inspector

The third inputType that grows a dataTransfer is insertReplacementText — fired when the OS accepts a spell-check or autocorrect suggestion. Type in the editor with deliberate misspellings, accept the suggestion, and watch the inspector show the strike-through diff and the carried-over original phrase.

Try typing teh and accepting Chrome's autocorrect. Or right-click a wavy red underline and pick a suggestion. The inspector receives the beforeinput, parses the diff via the new dataTransfer, and renders both versions inline as a track-changes view.

Try misspeling some werds heer. Chrome will suggest fixes — right-click an underline, pick one, and watch the diff land in the inspector.

What's happening

  1. The editor listens for beforeinput with inputType === "insertReplacementText".
  2. The new event.dataTransfer carries the suggested string in text/plain. The browser also exposes the old text via event.getTargetRanges()[0].
  3. Comparing the two gives a precise replacement diff — what the OS wants to write, what it's replacing, and where.
  4. Without this property, replacement detection required diffing the contenteditable's textContent after every event — fragile and lossy with IMEs.
  5. With track changes on, the editor cancels the default and inserts both versions wrapped in marks so the user can accept or reject.
editor.addEventListener("beforeinput", (e) => {
  if (e.inputType !== "insertReplacementText") return;
  const newText = e.dataTransfer?.getData("text/plain");
  const range   = e.getTargetRanges()[0];
  const oldText = range.toString?.() ?? "";
  showDiff(oldText, newText);
});
