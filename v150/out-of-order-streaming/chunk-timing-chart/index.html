why this matters

HTTP/2 and HTTP/3 multiplex streams and can deliver chunks out-of-order. Before Chrome 150, the browser's ReadableStream layer re-imposed an in-order constraint — if chunk 2 was delayed, chunks 3–6 sat in an internal buffer even though they had arrived. Chrome 150 removes that artificial serialisation. Useful work can begin the moment each chunk arrives, regardless of whether earlier chunks are ready.

// API is unchanged — the improvement is internal
const response = await fetch('/api/chunks');
const reader = response.body.getReader();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  // Chrome 150+: this fires as soon as *this* chunk is ready,
  // not after all preceding chunks have been flushed.
  processChunk(value);
}
