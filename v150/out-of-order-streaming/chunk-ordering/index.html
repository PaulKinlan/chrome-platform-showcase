Chunk Ordering

What head-of-line blocking looks like at the ReadableStream level, and how Chrome 150's out-of-order dispatch removes the bottleneck.

Chunks 3 and 4 delivered as soon as they arrive. Only chunk 2 itself is delayed — no blocking cascade.

performance impact

Metric Pre-Chrome 150 Chrome 150+
Time to first meaningful content Delayed if any early chunk is slow Available chunks render immediately
Time to complete stream Same (bottlenecked by slowest chunk) Same (bottlenecked by slowest chunk)
Server-sent events / SSE Re-ordering buffer adds latency Events dispatched as they arrive
LLM token streaming Stall when a chunk is re-transmitted Surrounding tokens still flow
HTTP/3 QUIC stream retransmit Retransmit blocks subsequent chunks Available chunks dispatched immediately

use cases

  • LLM token streaming — generated tokens arrive one at a time; a retransmitted packet no longer pauses the token display
  • Progressive HTML streaming — later sections of a page render without waiting for a slow earlier section
  • Server-sent events — real-time updates flow past any delayed event without queuing
  • Live data dashboards — chart updates continue while a stale data chunk is re-fetched

no API changes required

// The ReadableStream API is unchanged.
// Out-of-order dispatch is an internal browser improvement.
// Your existing reader code automatically benefits:

const response = await fetch('/api/tokens');
const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  // Chrome 150+: read() resolves as soon as ANY chunk is available,
  // not just the next expected chunk in sequence.
  appendToken(decoder.decode(value, { stream: true }));
}
