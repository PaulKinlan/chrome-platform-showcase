Streaming Comparison

Watch in-order streaming versus out-of-order streaming side by side. A slow section (900 ms) blocks everything below it in the in-order column. The out-of-order column shows each section as soon as it's ready — the slow section can't block the fast ones.

Chrome 150 required for the native declarative mechanism. This demo uses JavaScript timers to simulate both strategies visually.

Out-of-order streaming (Chrome 150)

why in-order streaming blocks

The HTML parser processes bytes sequentially. If the server starts sending the recommendations section (which needs a slow DB query), the parser can't move past it — bytes for the footer are not in the stream yet. The footer appears only after the slow section completes.

With out-of-order streaming, the server emits a <template for="recs"> placeholder immediately and streams the footer bytes right after. The actual recommendations arrive later via a <?start recs?> processing instruction — the browser inserts them into the placeholder without re-parsing everything between.

code comparison

<!-- In-order: server must wait for slow query before flushing footer -->
<nav>…</nav>
<h1>…</h1>
[... server waits 900 ms for DB query ...]
<section class="recs">…</section>  <!-- footer blocked here -->
<footer>…</footer>

<!-- Out-of-order (Chrome 150): placeholder emitted immediately -->
<nav>…</nav>
<h1>…</h1>
<template for="recs"><div class="skeleton">Loading…</div></template>
<footer>…</footer>  <!-- appears right away! -->
[... 900 ms later, DB query done ...]
<?start recs?><section class="recs">…</section><?end recs?>
