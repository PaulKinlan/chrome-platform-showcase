Out-of-order streaming

Chrome 150 enables out-of-order delivery for streamed HTTP responses — chunks from a ReadableStream body can be dispatched to consumers as soon as they arrive, even if earlier chunks in the sequence haven't been received yet. This removes a head-of-line blocking constraint within a single response stream.

why it shipped

With HTTP/2 and HTTP/3, the network layer can already deliver stream data out of order — but the browser's stream reading internals previously imposed an in-order constraint, re-buffering until the expected sequence was complete. This caused head-of-line blocking at the JavaScript ReadableStream level: a single delayed chunk stalled all subsequent chunks from being dispatched to the reader, even if they had already arrived. Chrome 150 removes this artificial serialisation, dispatching available chunks immediately.

the change

// Consuming a streaming response — unchanged API, improved delivery
const response = await fetch('/api/stream-data');
const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;

  // Chrome 150+: chunk dispatched immediately when it arrives,
  // no longer stalled behind a missing earlier chunk.
  // Head-of-line blocking within a single ReadableStream is removed.
  appendToUI(decoder.decode(value, { stream: true }));
}

// Also relevant: TransformStream and WritableStream pipelines
// benefit automatically — no code changes required.
