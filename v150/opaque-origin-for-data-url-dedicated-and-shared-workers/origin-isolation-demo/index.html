Origin Isolation Demo

Run the tests below to see how Chrome 150 changes the origin of workers created from data: URLs. In Chrome 149 and older, self.origin inside such a worker returned your page's origin — giving the worker same-origin access. In Chrome 150 it returns "null", blocking same-origin APIs.

Chrome 149 and older — inherited origin

A data: URL worker inherits the creator page's origin. self.origin returns the page's origin string. Same-origin APIs like BroadcastChannel and localStorage work normally.

What the test does

The demo creates a Dedicated Worker from a data: URL containing a small script. That script checks self.origin, then tries to create a BroadcastChannel, read from localStorage, and open an IndexedDB database — then posts the results back. In Chrome 150+ the worker's origin is opaque ("null"), which causes localStorage and IndexedDB to throw security errors since they require a non-opaque origin.

Test worker — created from a data: URL
// Worker source — stringified and turned into a data: URL const workerSrc = ` // What origin does this worker have? const origin = self.origin; // "null" in Chrome 150+ // Try BroadcastChannel (throws in opaque origins) let bcStatus; try { new BroadcastChannel('test'); bcStatus = 'accessible'; } catch (e) { bcStatus = 'blocked: ' + e.message; } // Try localStorage (not available in workers) let lsStatus = typeof localStorage !== 'undefined' ? 'accessible' : 'not in workers'; // Try IndexedDB (throws SecurityError in opaque origins) let idbStatus; try { indexedDB.open('test'); idbStatus = 'accessible'; } catch (e) { idbStatus = 'blocked: ' + e.message; } postMessage({ origin, bcStatus, lsStatus, idbStatus }); `; const url = 'data:text/javascript,' + encodeURIComponent(workerSrc); const worker = new Worker(url); // ← Chrome 150 gives opaque origin worker.onmessage = (e) => console.log(e.data);
