Fallback Experience

Capability elements degrade gracefully: when permission is denied or the API isn't supported, the element's visual state changes automatically. This demo shows all four states — idle, pending, granted, denied, and unsupported — and the fallback UI that should appear for each.

Without capability element (old approach)
Old pattern requires JS to:
1. Call getUserMedia()
2. Listen for devicechange
3. Query navigator.permissions
4. Manually grey out / show error
5. Handle browser differences

Capability elements do all of this with a single HTML attribute. The element's disabled/blocked state is reflected automatically through its CSS state.
Click "Enable Camera" above to trigger a real getUserMedia permission request, or use the simulate buttons to preview each state without touching the real API.
<!-- Capability element — browser manages state automatically --> <button type="capability" capability="camera"> Enable Camera </button> /* CSS for each state — no JS class toggling needed */ button[capability]:not([state]) { opacity: 1; } /* idle */ button[capability][state="granted"] { cursor: default; } /* granted */ button[capability][state="denied"] { opacity: 0.45; cursor: not-allowed; } button[capability][state="pending"] { background: yellow; }
