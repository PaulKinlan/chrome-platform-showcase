Rollout & controls

ServiceWorkerAutoPreload is a browser-internal optimisation: it ships enabled by default in Chrome 140 behind a stepped rollout, and there is deliberately no JS API that reports it. This page runs honest capability probes on the surfaces that do exist, explains what can and cannot be detected, and gives the exact switches that force the behaviour on or off.

Honesty note: the probes below read the exact service-worker surfaces this browser exposes. None of them can prove auto-preload is active — that requires the timing heuristic in the preload timing lab. A page that claimed a "live auto-preload probe" from JS alone would be lying to you.

force it on (any Chromium ≥ 131)

--enable-features=ServiceWorkerAutoPreload

force it off

--disable-features=ServiceWorkerAutoPreload

enterprise policy (admins)

ServiceWorkerAutoPreloadEnabled: false

This is a Chromium base::Feature, not a Blink runtime flag — so it's --enable-features, not --enable-blink-features, and there is no chrome://flags entry for it. The policy pins the behaviour for managed fleets regardless of the rollout stage.

stepped rollout, in plain words

Chrome 140's listing carries this feature twice: "Enabled by default" and "Stepped rollout". That means the default flipped in 140, but Google ramps the population percentage server-side — two Chrome 140 installs can genuinely behave differently on the same day. If a TTFB regression or origin traffic change appears in your metrics, don't bisect your own release; check whether the rollout reached your users, and use the switches above to reproduce both states locally.
