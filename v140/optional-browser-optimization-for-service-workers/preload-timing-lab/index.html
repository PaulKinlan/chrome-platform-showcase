Preload timing lab

This page registers a real service worker whose fetch handler adds a delay you control, then navigates a controlled iframe and reads the navigation's real PerformanceNavigationTiming. With ServiceWorkerAutoPreload (default since Chrome 140) the network request starts in parallel with the handler, so responseStart stops paying the handler-delay + network sum — the two legs overlap.

Run a measurement to see the verdict. The auto-preload behaviour has no direct JS probe — the honest signal is whether TTFB tracks max(handler delay, network) (overlapped) or handler delay + network (serialised). This is a heuristic inference, not a feature flag readout.

what the worker does

self.addEventListener("fetch", (event) => {
  const url = new URL(event.request.url);
  const mode = url.searchParams.get("mode");
  const delay = Number(url.searchParams.get("delay") || "0");
  if (mode === "fallback") {
    // No respondWith(): the browser falls back to the network. With
    // ServiceWorkerAutoPreload the already-in-flight response is used directly.
    return;
  }
  event.respondWith((async () => {
    if (delay) await new Promise((r) => setTimeout(r, delay));
    // With auto-preload this fetch() consumes the parallel request
    // instead of starting a new one after the delay.
    return fetch(event.request);
  })());
});
