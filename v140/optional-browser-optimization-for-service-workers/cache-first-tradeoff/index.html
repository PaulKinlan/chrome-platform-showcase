Cache-first trade-off

Auto-preload's one cost: if your fetch handler answers from CacheStorage, the parallel network request the browser fired during worker bootstrap is wasted bytes — the handler never consumes it. This page registers a real cache-first worker and fetches two probes through it: one answered from CacheStorage (never touches the network) and one passed through to the real server file (consumes the preload). the trade-off, precisely

The explainer is explicit that auto-preload targets handlers that end up needing the network (pass-through, network-first, fallback). For pure cache-first handlers the preload request is redundant: the response arrives, nobody reads it, and the origin has served bytes for nothing. That's why the feature shipped with a stepped rollout and an enterprise off-switch (ServiceWorkerAutoPreloadEnabled) rather than as an unconditional behaviour — and why Chrome pairs it with browser-side heuristics on when to preload at all. If your app is aggressively cache-first, measure your origin traffic before and after Chrome 140.

// cache-first: the preloaded response is never consumed → wasted request
self.addEventListener("fetch", (event) => {
  event.respondWith(caches.match(event.request).then((hit) => hit || fetch(event.request)));
});

// pass-through: the preloaded response IS consumed → pure win
self.addEventListener("fetch", (event) => {
  event.respondWith(fetch(event.request));
});
