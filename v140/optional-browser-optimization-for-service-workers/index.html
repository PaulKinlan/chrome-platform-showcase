v140 · service worker

Optional browser optimization for Service Workers

ServiceWorkerAutoPreload reaches stable: from Chrome 140 the browser issues the network request in parallel with service worker bootstrap on navigations, by default. If the fetch handler calls fetch(event.request) inside respondWith(), the in-flight response is consumed there; if the handler falls back (no respondWith()), the parallel response is handed straight to the page. These demos register real service workers and measure the effect with real PerformanceNavigationTiming entries — no simulated numbers.

concepts

  1. Preload timing lab

    Register a real service worker whose fetch handler adds a tunable delay, then reload a controlled page and read the real workerStart / fetchStart / responseStart numbers. Compare respondWith(fetch(…)) against the no-respondWith fallback path.

  2. Cache-first trade-off

    A cache-first handler answers from CacheStorage while auto-preload still puts a request on the wire. Measure the instant cache response, then see why the spec work is careful about wasted requests — and when the preload is consumed instead.

  3. Rollout & controls

    Enabled-by-default plus stepped rollout: probe what this browser exposes, and get the exact --enable-features / --disable-features switches and the ServiceWorkerAutoPreloadEnabled enterprise policy admins use to pin the behaviour.

why it shipped

Service worker fetch handlers sit on the critical path of every controlled navigation. When the worker is cold, the page pays worker bootstrap before the first byte is requested — a tax that hits pass-through and network-first handlers hardest. Chrome 131 trialled ServiceWorkerAutoPreload as an origin-trial-era browser mode (see the v131 demos for the timeline visualisations); Chrome 140 turns it on by default with a stepped rollout, so the network leg now overlaps worker bootstrap instead of queueing behind it. The optimisation is transparent to correct handlers: the preloaded response is deduplicated into fetch(event.request) or used directly on fallback.
