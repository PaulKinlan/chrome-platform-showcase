SpeechRecognitionResult Timestamps (WebSpeech API)

The Web Speech API used to be a black box about when in the audio a phrase was spoken. Chrome 153 adds two nullable attributes — audioStartTime and audioEndTime — to SpeechRecognitionResult, letting you line transcripts up with a media timeline and measure on-device recognition latency. These demos call the real Web Speech API and read the new fields, degrading honestly when your engine returns null.

concepts

  1. On-device latency monitor

    Compute processing lag as event.timeStamp − result.audioEndTime and decide, live, whether an app should fail over from the on-device engine to a cloud backend.

  2. Caption timeline mapper

    Place each recognised phrase on an audio timeline using audioStartTime and audioEndTime, the way a captioning or transcript-scrubbing tool would.

  3. Result-object capability probe

    Detect the timestamps correctly: SpeechRecognitionResult has no constructor to probe, so you must inspect a live result instance. Covers the nullable contract and the 2 ms fuzzing mitigation.

why it shipped

Two problems drove the proposal. Timeline association: without timestamps you cannot map a transcript back to the audio, which makes captioning, transcript scrubbing and highlight-as-you-play impossible. Latency tracking and failover: as recognition moves on-device for privacy, performance now depends on the visitor's hardware. Exposing audioEndTime lets an app compare it against the result event's timeStamp to detect when the local model is falling behind real time and seamlessly switch to a cloud backend before captions visibly lag.

the shape

partial interface SpeechRecognitionResult {
  readonly attribute DOMHighResTimeStamp? audioStartTime;  // ms, relative to time origin
  readonly attribute DOMHighResTimeStamp? audioEndTime;    // ms, relative to time origin
};

Both attributes are nullable: if the recognition backend does not supply segment timing, they are null — not an error. To limit fingerprinting, exposed values are fuzzed to 2 ms precision (matching HTMLMediaElement.currentTime).

enabling it

This is a developer-trial addition. Launch Chrome 153+ with --enable-experimental-web-platform-features (the catch-all that turns on in-development Web Speech attributes), and use a recognition backend that reports segment timing — on-device recognition needs its speech model downloaded. When the attributes are absent or null, the demos say so plainly and offer a clearly-labelled sample walkthrough instead of faking values.
